webpack不仅仅是打包工具

2018-10-06
  1. 什么是webpack?
    官网的解释：webpack是现代javascript应用中的一种静态模块打包机。它对项目中涉及的各个模块进行映射，构建出一个依赖图表并打包成一个或多个文件。由此，我们可以获知：第一，webpack主要是javascript文件的打包机；第二：webpack适用于模块化的js文件，构建依赖图表；第三：webpack最后打包成一个或多个文件。
    关于第一点，webpack可以借助各种loader进行转化，比如css-loader,babel-loader可以将css文件，es6文件进行转化。

  1. webpack配置文件
    webpack配置有很多的选项，本博客主要关注常用的内容，比如：entry,output,loader,plugin,resolve,mode,devserver，devtool等，分别为入口，输出，加载器，插件，模块解析，开发模式，开发工具（development/production）。目前webpack4.0.0最新版本中，webpack可以省略entry,output配置文件，当然，手动配置webpack更好。由于这块内容非常简单，不做细数。

  2. 加载器loader
    首先，我们思考一下为什么需要loader? 我们回到webpack的最初定义，它是用来打包js文件，因此我们需要使用loader对项目中涉及的许多静态资源(非js资源)，比如图片，字体，css等文件格式进行预处理，然后使之打包成为依赖图的一部分。
    常用的loader：
    styling(样式文件预处理）：css-loader,style-loader,less-loader,sacc-loader,postcss-loader
    file（文本文件预处理）：file-loader,url-loader
    ts/js转换：ts-loader/awesome-typescript-loader,babel-loader
    html预处理：html-loader

  3. 插件plugins
    提供文件打包优化功能，资产管理，环境变量的注入，与loader的转换功能不同，plugins的重点在于优化。 如html-webpack-plugin，text-extract-webpack-plugin等。webpack提供了非常丰富的plugin接口，根据项目的需要进行相应的配置，使得webpack的功能非常的强大。
    常用的plugins：
    CommonsChunkPlugin 提取多个入口文件的公共部分（区别于DllPlugin的提取），这样一来，页面首次该文件只需加载编译一次，缓存放在客户端，优化了页面加载速度。在webpack4版本中，该插件被splitChunksPlugin替换，实现了更好的优化功能
    DllPlugin和DllReferencePlugin 将依赖的第三方库（比如angular,jquery等）独立打包成dll文件，因为这些库文件在开发中基本不会去改动，避免每次编译，将它独立出来，可以提高页面加载的速度
    ExtractTextWebpackPlugin 将css文件从打包文件中独立出来
    HotModuleReplacementPlugin 模块热替换，开发人员在web应用运行时，修改，增加或者删除部分源码模块内容之后，浏览器进行相应的内容更新但不需要重新加载页面。
    HtmlWebpackPlugin 将打包后的文件通过script标签添加到html文件中
    SourceMapDevToolPlugin 在控制台source栏源代码可视化，适用于开发环境
    CopyWebpackPlugin 拷贝文件或文件夹到最终的打包文件夹

  4. mode
    webpack分开发环境和生产环境，相应调用内置的优化工具。可通过两种方式进行配置

  5. resolve
    resolve可以设置webpack如何寻找模块文件，比如通过更改别名来加载绝对路径的模块；

  1. devServer
    通过使用 webpack-dev-server,提供一个简单的服务器功能，npm run start启动后就会自动打开浏览器，并且修改保存源码后，服务器会重新编译，重新加载，简化了在开发调试操作上的繁琐。支持HotModuleReplacementPlugin热加载，相互合作。

  2. devtool
    该选项主要是通过使用SourceMapDevTooPlugin插件，可以在调试阶段，通过浏览器的控制台具体定位代码编译错误的位置

  3. why webpack?
    说了这么多配置，那么，webpack到底好在哪儿呢？ 除了webpack，还有很多其他的打包工具，比如grunt，gulp等，为什么webpack更受欢迎呢？
    我们知道js的引用有两种：一种就是在html文档中以script标签一个个地引用js脚本；一种就是将所有js脚本合并为一个js文件，在html文档中以script标签引用。前者的问题是导致网络请求太多，后者问题是一个js文件太大，代码不好维护。
    前期，我们通过IIFE’s(立即执行函数表达式)这种模块化的场景来解决上述的问题，因此grunt, gulp等工具诞生了，但它们只是简单地将所有模块串起来。当你改动了一部分的js代码，或者应用只需要部分文件时，这些工具依然需要重新对所有文件进行打包，也就是说效率非常低下。
    而ECMAScrip6中倡导的模块化开发中，webpack非常好地避开了这些问题，提高了打包的效率。

docker相关知识

2018-08-09

  1. 什么是docker？
    NodeJs解释：将应用程序及其依赖以一种标准的单位进行打包。换言之，docker是一个快速部署的轻量级虚拟技术项目，开发人员可以一起打包程序和运行环境，成为一个镜像，只需要下载镜像文件就可以将程序跑起来，免去每次都安装各种依赖和环境的麻烦，还能够做到应用程序之间的隔离。粗糙地认为docker类似于虚拟机，但不同于虚拟机。两者的共同点：隔离和配置计算机资源；不同点，docker是在操作系统层面的虚拟化，而虚拟机是在硬件层面的虚拟化,下列图片很好地展示了两者区别

    docker特点：1. 安全 2. 轻量 3. 标准化

  2. docker涉及的主要概念：docker images，docker container，docker registry
    docker images是文件系统和参数的组合，docker里面所有文件都是基于镜像文件存在的，它是一种轻量，独立可执行的软件，内部包含了该应用程序运行所需的一切文件（如code, runtime, system tools, system libraries and settings），后续我们举例nginx镜像文件进行详细说明。
    docker container是将所有程序及其依赖打包后的一种标准的单元，应用程序通过此容器，可以在不同的环境之间保持快速可靠运行
    docker registry是一种无状态，高容量的服务器端应用，用来存储和分配docker images文件

  3. 安装和配置： 官网（http://www.docker.com/）下载对应的docker，并按照指示进行安装。为了加速连接，提高docker的下载速度，可以添加加速器（可采用阿里云或者网易云的docker加速器），具体操作如下图

  4. 安装好之后，就可以在命令行进行docker基本操作了，其指令和linux的操作类似，只是多了关键字docker,对于了解linux指令的开发人员来说，掌握基本指令其实是非常简单的事情。下列通过下载使用nginx进行详细说明

    1. 查看容器的基本情况
      1.1 docker images //查看镜像列表
      1.2 docker ps //查看当前运行中的容器信息
      1.3 docker ps -a //查看所有容器信息，包括运行中和暂停的容器信息

    2. 下载nginx镜像文件并创建容器
      2.1 docker pull nginx //下载nginx镜像文件
      2.2 docker run -itd -p 10090:80 –name “my-nginx” nginx
      //创建并启动nginx的新容器，并把虚拟机的80端口映射到本机的10090端口，在浏览器中可通过localhost:10090进行访问到nginx
      2.3 docker stop #containerId //停止指定的容器，通过docker ps -a 查看被停止的容器信息
      2.4 docker start #containerId //重启指定的容器，通过docker ps/docker ps -a 查看容器的

    3. 在运行的容器内执行命令 ，语法： docker exec [OPTIONS] CONTAINER COMMAND [ARG…]
      3.1 docker exec -it #containerId bash //在运行的容器中执行bash命令，成功进入后就可以使用bash命令进行相应操作
      3.2 cat /etc/nginx/conf.d/default.conf //如查看nginx镜像文件的配置文件
      3.3 vim /usr/share/nginx/html/index.html //比如编辑nginx下的html文件（apt-get update更新软件包，apt-get install vim安装vim）

    4. 由于docker容器内文件的修改是暂存的，需要进行提交保存。比如上述步骤3.3修改后的镜像文件进行提交保存
      4.1 docker ps //查看
      4.2 docker diff #containerId //文件的修改内容对比
      4.3 docker commit -m “comment” #containerId nginx2:1.0.1
      通过docker images查看镜像列表，新添加了一个镜像nginx2

    5. 操作完了，我们可以删除容器或删除镜像
      5.1 dokcer stop containerId //首先停止运行中的容器
      5.2 docker rm #containerId //删除容器
      5.3 docker rmi #imageId //删除容器内部的镜像文件
      注意：如果没有执行前两步操作，直接删除镜像文件会失败，提示先停止该镜像所在容器，再删除镜像，若镜像被多个容器引用，需要强制删除,指令后面再加 –force

  5. 总之，对于开发者来说，使用docker可以更方便地搭建和部署应用软件，在独立的容器内进行操作，用完后删掉容器和相应的镜像文件，不影响本地。后续文章将讨论如何利用dockerNodeJs应用,敬请期待。
